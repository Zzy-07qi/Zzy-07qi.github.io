State

state 是 React 组件的内置对象，用于存储组件自身需要维护的动态数据。当 state 发生变化时，React 会重新调用组件的 render 方法（函数组件中则是重新执行组件函数），更新视图。

核心特点：

  • 私有性：仅属于当前组件，外部无法直接修改
  • 可变性：通过特定方法（ setState 或 useState 钩子）修改
  • 响应式：状态变化自动触发视图更新

函数组件中的 State：useState 钩子

在函数组件中，我们使用 React.useState 钩子来声明状态。

useState 接收一个参数作为状态的初始值（可以是任意类型：数字、字符串、对象、数组等）

  • 返回值是一个数组，包含两个元素：当前状态值和状态更新函数

  • 状态更新函数（如 setCount ）用于修改状态，调用后会触发组件重新渲染

State 进阶：更新机制与注意事项

状态更新的异步性

核心要点：React 中的状态更新（无论是 setState 还是 useState 的更新函数）是异步的。这意味着，在调用更新函数后，无法立即获取到最新的状态值。

解决方案：使用函数式更新

如果新状态依赖于旧状态，建议使用函数式更新，此时 React 会保证传入的函数参数是最新的状态值。

复杂状态的更新：对象与数组

当状态是对象或数组时，更新时需要注意不可变性——不能直接修改原对象/数组，而是要返回一个新的对象/数组。

常见场景与实践

状态提升：共享状态的管理

当多个组件需要共享同一个状态时，需要将状态提升到它们的最近共同父组件中，通过 props 将状态和更新函数传递给子组件。

局部状态与全局状态的划分

  • 局部状态：仅属于单个组件的状态（如输入框的输入值、按钮的禁用状态），使用 useState 管理。

  • 全局状态：多个组件共享的状态（如用户信息、主题设置），建议使用 useContext + useReducer 或状态管理库（如 Redux、Zustand）。

实践总结

  1. 保持状态最小化：只存储必要的动态数据，避免冗余状态

  2. 遵循不可变性：更新对象/数组时，始终返回新值

  3. 避免过度使用状态：可通过计算得到的数据（如列表的筛选结果），无需存储为状态

  4. 状态提升：多个组件共享状态时，提升到共同父组件

  5. 使用函数式更新：当新状态依赖于旧状态时，优先使用函数式更新

常见坑点与避坑指南

直接修改状态不触发渲染

问题：直接修改状态值（如 count = count + 1 ），不会触发组件重新渲染。
原因：React 无法检测到直接的状态修改，必须通过更新函数（ setCount ）修改。
解决方案：始终使用状态更新函数修改状态。

异步更新导致的状态不一致

问题：在调用更新函数后，立即获取状态值，得到的是旧值。
原因：状态更新是异步的，React 会批量处理更新以提高性能。
解决方案：使用函数式更新，或在 useEffect 中监听状态变化。

复杂状态的浅拷贝问题

问题：更新嵌套对象时，只做了浅拷贝，导致深层属性变化不触发渲染。
原因：React 比较状态是否变化时，使用的是浅比较（引用比较）。
解决方案：对嵌套对象进行深拷贝，或使用 Immer 库简化不可变更新。

运行 npm install use-immer 添加 Immer 依赖
用 import { useImmer } from ‘use-immer’ 替换掉 import { useState } from ‘react’

对比

忘记状态的初始值类型

问题：状态初始值为 undefined ，后续更新时类型不匹配，导致报错。
原因： useState 的初始值决定了状态的类型，后续更新应保持类型一致。
解决方案：始终为状态设置明确的初始值，避免 undefined 。
